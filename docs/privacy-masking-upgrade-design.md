# ClaimCraft 隐私识别、人工复核与安全导出升级设计方案

> 文档状态：建议方案  
> 适用范围：隐私打码页、证据图片访问、敏感信息识别、人工遮罩、脱敏任务、导出中心  
> 设计日期：2026-07-19  
> 设计原则：安全默认、自动建议而非自动担保、原始证据与分享副本分离、人工可复核、全链路可追溯、渐进式落地  
> 视觉约束：沿用现有绿色、米白、深墨色工作阶段页风格；不使用 Emoji；使用 `lucide-react` 图标；状态不得只依赖颜色表达

---

## 1. 背景与问题定义

ClaimCraft 当前已具备文本敏感信息扫描、图片自动遮盖、原图与打码图对比以及文本脱敏预览等基础能力。现有实现适合作为功能 Demo，但仍不足以向用户表达“材料已经安全脱敏”，主要原因如下：

1. 文本扫描当前主要覆盖证据描述，尚未统一扫描 OCR 全文、抽取字段、时间线、物证说明和最终文书；
2. 文本规则实际支持手机号、身份证号和结构化地址，页面却包含“姓名”等尚未真正落地的能力描述；
3. 图片打码独立使用 Tesseract 重新 OCR，没有复用主工作流中更丰富的 OCR 结果和坐标；
4. 自动打码后只能查看，无法新增、删除或调整遮罩，也没有明确的用户复核确认；
5. 批量图片处理同步执行，失败项、卡住的 `pending` 状态和单图重试缺少完整恢复路径；
6. 原始图片通过通用媒体路径提供，缺少证据所有权鉴权，且使用公开长期缓存策略；
7. 正式 PDF/Word 按当前产品要求始终使用原图，ZIP 同时包含原图和已有打码图，但隐私打码页没有向用户解释打码结果与各导出格式的关系；
8. 自动识别完成、人工复核完成和可用于隐私导出三个概念没有区分。

因此，本方案将当前“隐私打码”能力升级为：

> 一套面向案件材料的隐私风险扫描、自动遮罩建议、人工复核确认、衍生文件生成和安全导出体系。

系统不得把“算法没有发现风险”直接描述为“材料绝对安全”，也不得把“自动生成了打码图”直接等同于“用户已确认可对外分享”。

## 2. 升级目标与非目标

### 2.1 核心目标

1. **覆盖完整**：统一扫描案件中可能进入交付物的文本与图片内容；
2. **结果准确**：结构化规则、OCR 坐标、语义识别和视觉检测分层协作；
3. **人工可控**：用户可查看、增删、调整和确认遮罩区域；
4. **状态清晰**：明确区分未扫描、扫描中、待复核、已确认、需更新和失败；
5. **文件安全**：原图受案件权限保护，打码图作为独立衍生版本管理；
6. **导出明确**：原始证据版和隐私保护版具有不同且清晰的文件策略；
7. **可恢复**：任务可异步执行、刷新恢复、失败重试和超时接管；
8. **可追溯**：记录识别来源、遮罩版本、用户确认和导出策略；
9. **渐进兼容**：保留现有原图 PDF/Word 和 ZIP 能力，不在第一阶段破坏已有导出契约。

### 2.2 非目标

本方案不承诺：

- 自动识别达到零漏检；
- 用打码结果替代原始证据存档；
- 在未经过人工复核时提供“绝对安全”保证；
- 第一阶段同时支持视频、音频或多页扫描 PDF；
- 第一阶段删除用户上传的原始图片。

## 3. 产品术语与边界

为避免页面概念混淆，统一使用以下术语。

| 术语 | 定义 | 不应表达为 |
|---|---|---|
| 隐私扫描 | 系统对文本和图片查找潜在敏感信息 | 已完成安全脱敏 |
| 自动建议 | 系统根据规则和模型生成的候选敏感项或遮罩 | 已经人工确认 |
| 遮罩区域 | 图片中需要覆盖的矩形或多边形区域 | OCR 文本本身 |
| 脱敏副本 | 根据原图和已确认遮罩重新渲染的衍生图片 | 原始证据 |
| 人工复核 | 用户检查并确认文本项和图片遮罩 | 算法自动完成 |
| 原始证据版 | 保留用户上传原图的归档或受控提交材料 | 隐私分享材料 |
| 隐私保护版 | 仅使用已确认脱敏文本和脱敏图片的分享材料 | 证据原件 |
| 需更新 | 原图、OCR、文本或策略变化后，旧复核结果已失效 | 仍可安全导出 |

页面主标题可继续使用“隐私打码”，但页面说明应使用“识别建议”“复核”“确认”等准确表述。

## 4. 总体业务流程

```text
案件材料
  │
  ├─ 文本来源收集
  │    ├─ 证据描述 / OCR 全文 / OCR 摘要 / 物证说明
  │    ├─ 抽取字段 / 时间线
  │    └─ 最新投诉书或反证答辩书
  │
  ├─ 图片与 OCR 坐标收集
  │
  ▼
创建隐私扫描任务
  │
  ├─ 强规则识别：手机号、身份证、邮箱、银行卡等
  ├─ 上下文与 NER：姓名、地址、账号等
  ├─ 视觉检测：人脸、二维码、车牌等
  └─ 合并、去重、置信度与风险分级
  │
  ▼
生成自动建议
  │
  ├─ 文本候选项
  └─ 图片遮罩候选区域
  │
  ▼
人工复核
  │
  ├─ 接受 / 忽略文本项
  ├─ 新增 / 调整 / 删除图片遮罩
  └─ 确认本轮复核
  │
  ▼
生成脱敏副本
  │
  ├─ 不透明遮盖或不可逆像素化
  ├─ 移除 EXIF / GPS 元数据
  └─ 保存版本、来源哈希和确认记录
  │
  ▼
导出安全门
  ├─ 原始证据版：允许使用原图，强提示包含未脱敏信息
  └─ 隐私保护版：仅使用已确认且未过期的脱敏版本
```

### 4.1 核心判断原则

- `扫描完成` 只表示系统执行完检测；
- `待复核` 表示已经有自动建议，但用户尚未确认；
- `已确认` 表示当前材料版本下用户已完成检查；
- `可隐私导出` 需要所有必需材料均为已确认且未过期；
- 原图或文本内容发生变化时，相关确认状态必须自动变为 `需更新`。

## 5. 敏感信息识别策略

### 5.1 扫描范围

统一扫描以下来源，避免只扫描 `Evidence.description`：

| 来源 | 字段示例 | 是否进入默认扫描 |
|---|---|---|
| 证据基础文本 | `description`、`physical_note` | 是 |
| OCR 结果 | `extracted_text`、结构化 OCR blocks | 是 |
| OCR 摘要 | `ocr_summary` | 是，但与 OCR 命中去重 |
| 抽取字段 | `ExtractedField.field_value` | 是 |
| 时间线 | `TimelineNode.event` | 是 |
| 最终文书 | 最新投诉书或答辩书正文 | 是 |
| 案件基础信息 | 标题、描述 | 可配置，默认扫描描述 |
| 文件元数据 | 文件名、EXIF/GPS | 是，分别检查和清理 |

扫描结果必须携带 `source_type`、`source_id`、`source_field` 和字符范围，前端才能准确展示来源并允许跳转核对。

### 5.2 敏感类型分层

#### 第一层：强结构规则

适合使用正则和校验算法，准确率高：

- 中国大陆手机号及 `+86` 形式；
- 身份证号，并执行生日合法性和校验位验证；
- 邮箱；
- 银行卡号，并使用长度、BIN 和 Luhn 校验降低误报；
- IP 地址；
- 护照号、统一社会信用代码等可配置规则。

#### 第二层：上下文与实体识别

需要结合上下文关键词、NER 或结构化字段：

- 姓名、收件人、联系人；
- 详细地址；
- 微信号、支付宝账号、平台账号；
- 车牌号；
- 单位或学校等可能构成身份关联的信息。

不得只依赖 LLM 输出。推荐由 NER/规则产生候选，LLM 仅参与低置信度语义判断，并记录检测方法。

#### 第三层：视觉敏感内容

- 人脸与头像；
- 二维码、条形码；
- 车牌；
- 手写姓名、签名；
- 图片角落中的定位、设备或账号水印。

### 5.3 白名单与证据保留

订单号、物流单号、平台交易号往往是案件证据，不应因“长数字”被无差别遮盖。系统需支持：

- 类型白名单；
- 字段上下文判断；
- 用户将误报标记为“保留”；
- 案件级策略复用；
- 导出前展示被保留但可能敏感的项目。

### 5.4 风险等级

| 等级 | 示例 | 默认动作 |
|---|---|---|
| 高 | 身份证、银行卡、人脸、二维码 | 默认建议遮盖，未复核阻断隐私导出 |
| 中 | 手机号、邮箱、详细地址、账号 | 默认建议遮盖，未复核阻断隐私导出 |
| 低 | 姓名、昵称、机构、城市级位置 | 建议复核，可按策略决定是否阻断 |
| 待判断 | 模型低置信度实体 | 必须人工判断，不得自动视为安全 |

## 6. OCR 与图片遮罩设计

### 6.1 复用统一 OCR 产物

打码服务不应再次独立运行 Tesseract 作为唯一坐标来源。主工作流 OCR 应输出并持久化：

```json
{
  "text": "13800138000",
  "polygon": [[120, 80], [280, 80], [280, 112], [120, 112]],
  "confidence": 0.97,
  "page": 1,
  "engine": "paddleocr-vl",
  "engine_version": "...",
  "source_image_hash": "sha256:..."
}
```

推荐优先级：

1. 使用主工作流已经生成的 OCR token/block 坐标；
2. 若现有 OCR 产物没有坐标，再执行专用坐标 OCR；
3. OCR 低置信度或无法定位时，进入人工复核，不生成“安全完成”状态；
4. Tesseract 可保留为本地回退，不再作为唯一生产方案。

### 6.2 精确区域定位

当前按整行遮盖容易丢失证据上下文。升级后应：

- 建立字符范围到 OCR token/polygon 的映射；
- 只合并实际命中实体覆盖的 token；
- 对跨行地址生成多个相邻区域；
- 对区域添加可配置安全边距；
- 对人脸、二维码等直接使用检测框；
- 保存归一化坐标，避免缩放后坐标失真。

### 6.3 遮罩渲染

默认使用不可逆的不透明遮盖，而非弱高斯模糊：

- 默认样式：深墨色不透明矩形；
- 可选样式：高强度像素化；
- 不建议：低强度高斯模糊；
- 每次从原图 + 当前确认区域重新生成，不叠加编辑旧打码图；
- 输出 PNG 或无损 WebP，避免 JPEG 压缩边缘伪影；
- 移除 EXIF、GPS、缩略图和其他元数据；
- 渲染后可执行二次 OCR 检查，但二次 OCR 只能作为提示，不替代人工复核。

### 6.4 图片完整性与版本失效

以 `source_image_hash` 标识原图版本。发生以下情况时，打码版本自动标记为 `stale`：

- 原图替换；
- OCR 版本变化并重新识别；
- 敏感识别策略变化；
- 用户新增或删除遮罩；
- 遮罩渲染参数变化。

## 7. 人工复核与遮罩编辑器

### 7.1 编辑器能力

建议在隐私页为单张证据提供全屏或大尺寸编辑器，支持：

- 图片缩放、平移、适配窗口；
- 展示/隐藏自动建议框；
- 选中遮罩并拖动、缩放；
- 新增矩形或多边形区域；
- 删除误报区域；
- 设置敏感类型与风险等级；
- 将候选项标记为“保留，不遮盖”；
- 撤销、重做；
- 原图、遮罩叠加、脱敏预览三种模式；
- 保存草稿；
- “已检查当前图片”确认动作；
- 键盘操作和可访问性说明。

React 技术选型可优先评估 `react-konva`，也可使用 Fabric.js 或 SVG。应避免自行实现复杂 Canvas 事件系统。

### 7.2 确认语义

用户确认时应显示：

> 我已检查自动建议和图片其他区域，确认当前遮罩可用于生成隐私保护副本。

确认记录至少包含：

- 用户 ID；
- 确认时间；
- 原图哈希；
- 遮罩版本；
- 识别策略版本；
- 区域数量；
- 接受、忽略和手工新增数量。

### 7.3 自动完成限制

仅在以下条件同时满足时，系统才可将图片标记为“自动检查完成”，但仍建议显示“建议人工复核”：

- OCR 与视觉检测均成功；
- 无低置信度项；
- 识别覆盖符合策略；
- 二次检查未发现残留；
- 产品策略明确允许自动通过。

默认方案仍应要求用户确认后进入“可隐私导出”。

## 8. 状态模型与异步任务

### 8.1 案件级扫描状态

```text
not_started     未开始
queued          已排队
scanning        扫描中
review_required 待复核
confirmed       已确认
stale           材料变更，需更新
partial_failed  部分失败
failed          扫描失败
```

### 8.2 单项状态

文本候选和图片均应具备独立状态：

```text
detected  自动识别
accepted  接受建议
ignored   确认保留
edited    人工调整
confirmed 已确认
stale     已失效
failed    处理失败
```

### 8.3 异步任务模型

建议新增 `PrivacyScanRun`，不要在一个 HTTP 请求内串行处理所有图片。

核心字段：

- `id`、`case_id`、`status`；
- `policy_snapshot`、`policy_version`；
- `total_items`、`completed_items`、`failed_items`；
- `started_at`、`heartbeat_at`、`finished_at`；
- `error_code`、`error_message`；
- `revision`；
- `created_by`。

创建任务后通过 SSE 或轮询更新。项目已有工作流 SSE 与快照架构，可复用“事件增量通知 + Snapshot 当前真相”的原则，但隐私任务应保持独立领域模型，避免混入法律分析工作流节点。

### 8.4 卡死与重试

- `scanning` 超过阈值且心跳过期时标记为 `stale/failed`；
- 支持重试全部失败项；
- 支持单张图片重试；
- 支持用户跳过自动识别并直接人工编辑；
- 重试必须幂等，不覆盖用户已确认的有效版本；
- 同一案件仅允许一个活动扫描任务，或明确采用版本隔离。

## 10. API 设计建议

### 10.1 扫描任务

```text
POST /api/cases/{case_id}/privacy-scan-runs/
GET  /api/cases/{case_id}/privacy-scan-runs/latest/
GET  /api/privacy-scan-runs/{run_id}/snapshot/
GET  /api/privacy-scan-runs/{run_id}/events/
POST /api/privacy-scan-runs/{run_id}/retry/
POST /api/privacy-scan-runs/{run_id}/cancel/
```

创建参数可包含：

```json
{
  "scope": ["text", "images"],
  "entity_types": ["phone", "id_card", "email", "address", "name", "face", "qrcode"],
  "address_level": "detail",
  "include_documents": true
}
```

### 10.2 复核操作

```text
GET   /api/privacy-scan-runs/{run_id}/entities/
PATCH /api/sensitive-entities/{entity_id}/
GET   /api/evidences/{evidence_id}/mask-versions/latest/
POST  /api/evidences/{evidence_id}/mask-versions/
PATCH /api/mask-versions/{version_id}/regions/
POST  /api/mask-versions/{version_id}/render/
POST  /api/mask-versions/{version_id}/confirm/
POST  /api/evidences/{evidence_id}/privacy-scan/retry/
```

区域更新建议一次提交完整区域集合并携带版本号，使用乐观锁避免多个页面互相覆盖。

### 10.3 导出检查

```text
POST /api/cases/{case_id}/privacy-export-check/
```

响应示例：


## 11. 文件访问与安全设计

### 11.1 取消通用公开证据媒体访问

用户证据不应继续依赖无鉴权的通用 `/media/...` 路径。建议改为：

```text
GET /api/evidences/{id}/files/original/
GET /api/evidences/{id}/files/masked/?version={version}
```

后端必须通过 `case__owner=request.user` 校验所有权。

如果使用对象存储：

- API 鉴权后返回短时签名 URL；
- 原图和脱敏图使用不同存储前缀和权限策略；
- 签名 URL 不进入长期日志；
- 禁止永久公开 Bucket。

### 11.2 缓存策略

| 文件 | 推荐缓存策略 |
|---|---|
| 原始证据 | `private, no-store` |
| 未确认脱敏预览 | `private, no-store` |
| 已确认脱敏副本 | `private, max-age`，并使用版本化 URL |
| 公共静态资源 | 可继续 `public, immutable` |

### 11.3 审计与日志

记录：

- 原图查看和下载；
- 隐私扫描任务创建；
- 遮罩新增、修改和删除；
- 用户确认；
- 隐私保护版导出；
- 管理员访问和异常权限尝试。

日志中不得写入完整身份证、手机号、地址或原图签名 URL。

## 12. 导出策略设计

### 12.1 保留现有原图导出契约

当前 PDF/Word 按产品要求使用原图，ZIP 包含原始素材并可附带已有打码图。本次升级不应静默改变这一行为，以免影响证据完整性和既有用户预期。

但必须在导出中心清楚标记：

- PDF：原始证据版，证据图片使用原图；
- Word：原始证据版，证据图片使用原图；
- ZIP：原始材料包，包含原图；若有脱敏图则作为附加副本；
- 文本预览：仅在开启选项时对文本进行脱敏。

### 12.2 新增隐私保护版

建议为 PDF、Word、ZIP 增加显式的材料版本选择：

| 版本 | 图片策略 | 文本策略 | 使用场景 |
|---|---|---|---|
| 原始证据版 | 原图 | 原文 | 个人归档、律师或机构受控提交 |
| 隐私保护版 | 仅已确认脱敏图 | 统一脱敏文本 | 咨询、演示、非必要接触方分享 |

隐私保护版必须通过导出安全门。存在以下情况时默认阻断：

- 图片待复核；
- 打码图失效；
- 图片自动处理失败；
- 文本有未处理的高/中风险实体；
- 最新文书在复核后发生变化。

### 12.3 ZIP 目录建议

原始证据版：

```text
manifest.json
complaint.txt
timeline.txt
images/original/
images/masked/       可选附加副本
```

隐私保护版：

```text
manifest.json
complaint_masked.txt
timeline_masked.txt
images/
  001_E1_masked.png
  002_E2_masked.png
```

隐私版不得混入原图。`manifest.json` 应记录策略版本、复核时间和遮罩版本，但不得包含敏感原文。

## 13. 前端信息架构

### 13.1 页面定位

现有 `/mask` 保持为独立工作阶段页，建议页面名称：

- 导航：`隐私打码`；
- 页面主标题：`隐私检查与打码`；
- 页面说明：`系统先识别可能的敏感信息并生成遮罩建议，请在对外分享前逐项复核。自动识别不能替代人工检查。`

不再使用“接口仅下发脱敏结果”等技术化描述作为主要用户文案。技术说明可放入帮助区。

### 13.2 页面结构

页面沿用其他阶段页的结构：

```text
深墨色阶段 Hero
  ├─ 阶段标识：隐私风险检查
  ├─ 标题：隐私检查与打码
  ├─ 准确的能力说明
  ├─ 案件标题 / 类型标签
  └─ 3 个摘要指标

状态与提示区
  ├─ 扫描状态条 / 进度
  ├─ 自动识别免责声明
  └─ 错误或需更新提示

主体：左主栏 + 右侧栏
  ├─ 左主栏
  │    ├─ 扫描范围与策略
  │    ├─ 文本风险清单
  │    └─ 图片复核清单
  └─ 右侧栏
       ├─ 当前检查概览
       ├─ 待处理事项
       └─ 导出影响说明

图片编辑器 Drawer / Dialog

底部完成区
  ├─ 完成条件清单
  └─ 确认本轮隐私复核
```

### 13.3 Hero 指标

建议只展示用户能理解且可执行的指标：

1. `待复核`：尚未确认的文本项 + 图片数；
2. `已确认图片`：当前版本已确认的图片数；
3. `高风险项`：身份证、银行卡、人脸等高风险未解决数量。

避免继续使用“涉及类型”作为核心指标，因为类型数量不能反映完成度。

### 13.4 页面阶段状态条

Hero 下方增加状态条：

| 状态 | 标题 | 说明 | 主操作 |
|---|---|---|---|
| 未开始 | 尚未扫描当前材料 | 扫描文本和图片中的潜在隐私信息 | 开始隐私扫描 |
| 扫描中 | 正在检查材料 | 显示已完成/总数，可刷新恢复 | 查看进度 |
| 待复核 | 已生成自动建议 | 仍有若干项需要人工确认 | 继续复核 |
| 已确认 | 本轮隐私复核已完成 | 当前材料可生成隐私保护版 | 前往导出 |
| 需更新 | 材料发生变化 | 旧确认不再覆盖最新内容 | 重新扫描 |
| 部分失败 | 部分材料未完成 | 失败项不能视为已安全处理 | 重试失败项 |

### 13.5 扫描策略区

首次扫描前以简洁卡片展示：

- 扫描范围：文本、图片；
- 默认类型：身份信息、联系方式、详细地址、账号与视觉隐私；
- 高级设置折叠区：姓名、城市级地址、头像、二维码等；
- 主按钮：`开始隐私扫描`；
- 辅助说明：`扫描只会生成建议，不会修改原始证据。`

不要使用“一键打码所有图片”作为唯一入口，因为该文案会让用户误以为处理完成后无需复核。

### 13.6 文本风险清单

建议按来源分组：

- 最新文书；
- 证据材料；
- 时间线；
- 抽取字段。

每一项展示：

- 类型和风险等级；
- 脱敏后的安全预览；
- 来源对象，如 `E3 · OCR 文本`；
- 检测方法和置信度（放在详情中）；
- `接受遮盖`、`保留原文`、`查看上下文`；
- 当前状态。

默认不在列表直接显示完整原文。查看上下文需要明确动作，并使用受控区域短暂显示。

### 13.7 图片复核清单

图片卡片不再只显示原图/打码图，而应突出任务状态和下一步：

```text
E1 订单截图                    待复核
自动建议 3 处 · 高风险 1 处
[缩略图 + 遮罩轮廓预览]
[查看并调整] [接受全部建议]
```

不同状态动作：

- `未扫描`：扫描此图；
- `扫描中`：显示进度，不重复提交；
- `待复核`：查看并调整；
- `已确认`：查看确认版本、重新编辑；
- `需更新`：重新扫描；
- `失败`：显示可读原因、重试、人工编辑。

### 13.8 图片编辑器布局

桌面端：

```text
┌────────────────────────────────────────────────────────┐
│ E3 隐私复核                原图 | 遮罩叠加 | 脱敏预览 │
├───────────────────────────────────┬────────────────────┤
│                                   │ 检测到的区域       │
│             图片画布              │ 1 手机号  高风险   │
│                                   │ 2 地址    中风险   │
│                                   │ + 添加遮罩         │
├───────────────────────────────────┴────────────────────┤
│ 缩放  撤销  重做            保存草稿  确认当前图片     │
└────────────────────────────────────────────────────────┘
```

移动端：全屏页面或底部抽屉，区域列表置于画布下方，不使用狭窄双栏。

### 13.9 右侧概览栏

保持与 `TimelinePage`、`ExportPage` 相同的约 `310–330px` 侧栏风格，展示：

- 检查进度；
- 待复核数量；
- 失败项；
- 原图与隐私版的区别；
- `PDF/Word 当前仍使用原图` 的明确说明；
- 前往导出按钮，仅在已确认时强调。

### 13.10 完成确认区

完成按钮不应叫“全部打码完成”，建议使用：

- 主按钮：`确认本轮隐私复核`；
- 完成后状态：`已确认，可生成隐私保护版`。

按钮上方显示条件：

- 所有高风险文本项已处理；
- 所有图片已检查；
- 无失败或过期材料；
- 用户已阅读自动识别说明。

## 14. 前端视觉与交互规范

### 14.1 与现有工作阶段页保持一致

### 14.2 状态视觉

每个状态同时提供图标、文字和颜色：

| 状态 | 图标建议 | 颜色倾向 |
|---|---|---|
| 未开始 | `ScanSearch` | 中性灰 |
| 扫描中 | `Loader2` | 蓝绿或辅助绿 |
| 待复核 | `CircleAlert` | 米黄 |
| 已确认 | `ShieldCheck` | 绿色 |

| 失败 | `TriangleAlert` | 红色 |
| 人工调整 | `PenTool` | 深墨色 |

不得只用绿色/红色表达结果；所有图标都需要可见文本或 `aria-label`。

### 14.3 文案规范

推荐：

- `发现 6 个可能的敏感项`；
- `自动建议，待人工复核`；
- `已检查当前图片`；
- `材料发生变化，需要重新检查`；
- `未发现不代表绝对不存在，请在分享前核对原图`。

避免：

- `100% 安全`；
- `已完全脱敏`（除非明确指用户确认的版本）；
- `AI 已确保无隐私泄露`；
- `处理成功`但不说明是否待复核；
- `原始内容不下发`等面向实现的技术文案。

### 14.4 加载、空状态和错误

必须区分：

- 正在加载历史扫描结果；
- 尚未开始扫描；
- 扫描完成且未发现候选项；
- 扫描失败；
- 材料为空；
- 结果已过期。

“未发现候选项”应显示：

> 本轮自动扫描未发现匹配项。自动识别可能存在遗漏，请在对外分享前检查图片和最终文书。

### 14.5 可访问性

- 画布编辑必须提供区域列表作为等价操作入口；
- 支持键盘选择、移动、删除遮罩；
- 对话框正确管理焦点并支持 Escape；
- 状态更新使用 `aria-live="polite"`；
- 进度使用 `role="progressbar"`；
- 原图查看动作需要明确告知可能包含敏感信息；
- 移动端触控目标不小于 44px。

## 17. 性能、并发与幂等

### 17.1 性能

- OCR 结果按原图哈希缓存；
- 文本扫描按字段内容哈希缓存；
- 图片检测允许有限并行，避免单请求串行；
- 大图先生成编辑预览图，但渲染最终副本必须使用原始分辨率；
- 前端缩略图和编辑器资源分开加载；
- 区域列表虚拟化仅在数量较大时引入。

### 17.2 并发

- 同一案件活动扫描任务使用唯一约束或幂等 key；
- 区域保存使用 `revision` 乐观锁；
- 渲染任务由 `source_hash + regions_hash + render_policy` 构成幂等键；
- 用户编辑期间若原图变更，阻止确认并提示重新加载；
- 不再通过修改 `pytesseract` 全局命令路径处理并发配置。

### 17.3 失败策略

- 单图片失败不终止整个任务；
- 失败原因分为环境、文件、OCR、检测、渲染和权限；
- API 返回稳定错误码，不只返回通用中文字符串；
- 前端提供对应恢复动作；
- 无法自动识别时允许进入人工编辑，而不是陷入死状态。

## 18. 测试与验收标准

### 18.1 单元测试

文本检测至少覆盖：

- 手机号、`+86`、数字边界；
- 身份证格式、生日、校验位；
- 邮箱；
- 银行卡 Luhn 和长订单号误报；
- 结构化与非结构化地址；
- 姓名上下文；
- 空值和超长文本；
- 多类型重叠和去重。

图片服务至少覆盖：

- OCR token 到实体区域映射；
- 同行局部命中而非整行遮盖；
- 跨行地址；
- 区域边界裁切；
- PNG/WebP 无损输出；
- EXIF/GPS 移除；
- 原图哈希变化后版本失效；
- 重试、并发与幂等。

### 18.2 API 与权限测试

- 未登录无法访问原图和脱敏图；
- 用户 A 无法访问用户 B 的证据；
- 原图响应不得使用 public 缓存；
- 过期确认不能通过隐私导出检查；
- 隐私 ZIP 不包含原图；
- 原始版导出仍按既有契约工作；
- 审计日志不记录敏感原文。

### 18.3 前端测试

- 加载、未开始、空结果、失败、需更新状态正确区分；
- 扫描进度可刷新恢复；
- 单图重试和人工编辑入口可用；
- 遮罩新增、调整、删除、撤销和保存；
- 未确认时隐私导出被阻断；
- 原始版导出警告准确；
- 桌面、平板和移动端布局；
- 键盘操作、焦点和读屏状态更新。

### 18.4 核心验收场景

1. 用户上传包含手机号和地址的聊天截图，系统给出局部遮罩建议，用户可调整后确认；
2. 系统遗漏一处昵称，用户可手动新增遮罩并重新渲染；
3. 某图片 OCR 失败，页面明确显示失败原因并允许人工编辑；
4. 用户修改文书后，隐私状态变为“需更新”，隐私版导出被阻断；
5. 用户选择原始证据版 PDF，页面明确提示使用原图且不受打码图影响；
6. 用户选择隐私保护版 ZIP，压缩包中不出现任何原图；
7. 未登录或其他用户无法通过旧媒体链接读取证据；
8. 扫描任务中断后可通过超时恢复或重试继续，不永久停留在处理中。

## 19. 分阶段实施计划

### 阶段 A：安全与文案纠偏（P0）

目标：先消除误导和直接文件风险，不改变复杂业务模型。

- 证据媒体改为鉴权访问；
- 原图改为 `private, no-store`；
- 修正页面姓名等未实现能力描述；
- 增加初始加载、失败和空状态；
- 明确 `/mask` 是自动建议 + 人工检查；
- 导出中心明确现有 PDF/Word/ZIP 原图策略；
- 增加单图重试和 `pending` 超时恢复；
- 为现有图片打码服务补单元测试。

### 阶段 B：统一扫描与异步任务（P1）

目标：扩大覆盖范围并解决同步批处理问题。

- 建立 `PrivacyScanRun` 和 Snapshot；
- 扫描 OCR、抽取字段、时间线和最终文书；
- 复用主工作流 OCR 坐标；
- 增加邮箱、银行卡、姓名上下文等类型；
- 支持进度、刷新恢复和失败项重试；
- 建立统一状态模型和 `privacy-store`。

### 阶段 C：人工遮罩编辑与确认（P1）

目标：形成真正的人工复核闭环。

- 新增实体、遮罩区域和版本模型；
- 实现图片遮罩编辑器；
- 支持接受、忽略、手工新增和调整；
- 保存确认记录和审计；
- 原图变化后自动失效；
- 生成无损且移除元数据的脱敏副本。

### 阶段 D：双轨导出（P1）

目标：让打码结果真正服务于安全分享，同时保留原始证据能力。

- 新增原始证据版 / 隐私保护版；
- 新增隐私导出安全门；
- PDF、Word、ZIP 支持隐私保护版；
- 隐私 ZIP 严禁包含原图；
- manifest 记录策略与版本；
- 导出失败可直接跳转到待复核项。

### 阶段 E：高级视觉隐私与策略（P2）

- 人脸、头像、二维码、车牌和签名检测；
- 案件级与用户级隐私策略；
- 多引擎交叉验证；
- 批量接受低风险建议；
- 管理端质量指标和误报/漏报反馈闭环。

## 20. 迁移与兼容策略

1. 保留现有 `/cases/{id}/mask/` 和 `/mask-images/`，前端迁移完成前不删除；

5. 当前 PDF/Word 原图行为保持不变，并在页面准确说明；
6. 新增隐私保护版而非静默替换原始证据版；
7. 新路径端到端验证通过后，再移除旧 `masked` / `toggleMasked` 和旧页面逻辑。

## 21. 风险与取舍

| 风险 | 影响 | 缓解方式 |
|---|---|---|
| 人工复核增加操作成本 | 用户完成路径变长 | 风险分级、批量接受、只强制高/中风险 |
| NER/视觉模型误报 | 遮盖证据关键信息 | 保留建议状态、允许忽略和恢复 |
| 漏报仍不可避免 | 隐私泄露 | 明确免责声明、人工编辑、导出安全门 |
| 数据模型复杂度增加 | 开发与迁移成本 | 分阶段落地、兼容旧字段 |
| 异步任务增加状态复杂度 | 前后端联调成本 | Snapshot 为真相、SSE 仅做增量 |
| 原图鉴权影响图片加载 | 性能和缓存变化 | 短时签名 URL、私有版本缓存 |
| 双轨导出增加选择成本 | 用户可能选错 | 使用场景说明、默认推荐、强确认 |

## 22. 最终建议

本次升级不应继续以“增加更多正则”或“换一个 OCR 引擎”作为唯一方向，而应同时完成四个层面的调整：

1. **产品语义**：从“一键完成打码”改为“扫描建议—人工复核—确认—安全导出”；
2. **技术链路**：复用统一 OCR 坐标，建立敏感实体、遮罩版本和异步任务；
3. **安全边界**：原图鉴权、私有缓存、访问审计和隐私导出安全门；
4. **前端体验**：准确状态、可编辑遮罩、清晰导出影响和与其他阶段页一致的视觉结构。

完成阶段 A 后，当前页面可被准确定位为“隐私风险扫描与预览”；完成阶段 C 和 D 后，才具备较完整的“人工确认脱敏与隐私保护版导出”能力。
